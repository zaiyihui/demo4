# 项目结构维护指南

---

## 1. 目的与范围

本指南旨在规范 Computer Companion 项目的文件组织结构，确保代码库保持清晰、可维护和易于扩展。

**适用范围**：
- 所有项目成员
- 新加入的开发人员
- 代码审查和重构活动

---

## 2. 目录结构规范

### 2.1 标准目录

| 目录 | 用途 | 内容类型 |
|------|------|----------|
| `Models/` | 数据模型 | 业务数据类、枚举、DTO |
| `Services/` | 核心服务 | 业务逻辑服务、数据处理 |
| `ViewModels/` | 视图模型 | MVVM 视图模型类 |
| `Views/` | 用户界面 | XAML 视图和代码后端 |
| `Assets/` | 静态资源 | 图标、图片、字体 |
| `docs/` | 项目文档 | 设计文档、用户手册 |
| `Tests/` | 测试代码 | 单元测试、集成测试 |

### 2.2 禁止目录

| 目录 | 原因 |
|------|------|
| `Common/` | 过于模糊，应按功能细分 |
| `Utils/` | 应命名为 Helpers 或 Utilities |
| `Lib/` | 依赖应通过 NuGet 管理 |
| `Debug/` | 临时文件，不应提交 |

## 3. 文件命名规范

### 3.1 C# 类文件

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| **模型类** | `{名称}Model.cs` 或直接命名 | `UserModel.cs` |
| **服务类** | `{名称}Service.cs` | `HardwareMonitorService.cs` |
| **视图模型** | `{视图名}ViewModel.cs` | `MainWindowViewModel.cs` |
| **枚举** | 在 Models 文件中定义 | - |
| **辅助类** | `{功能}Helper.cs` | `WindowHelper.cs` |

### 3.2 Avalonia 视图文件

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| **XAML 视图** | `{窗口名}.axaml` | `MainWindow.axaml` |
| **代码后端** | `{窗口名}.axaml.cs` | `MainWindow.axaml.cs` |

### 3.3 命名注意事项

- ✅ **推荐**：`MainWindowViewModel.cs`
- ❌ **避免**：`MainVM.cs`（缩写不清晰）
- ✅ **推荐**：`UserSettings.cs`
- ❌ **避免**：`Settings1.cs`（数字后缀）

## 4. 新增文件流程

### 4.1 添加新模型

1. 确认模型属于哪个功能领域
2. 在 `Models/` 目录创建文件
3. 命名遵循 PascalCase 规则
4. 添加适当的 XML 文档注释

```csharp
namespace ComputerCompanion.Models;

public class NewModel
{
    // 属性定义
}
```

### 4.2 添加新服务

1. 评估服务职责是否单一
2. 在 `Services/` 目录创建文件
3. 命名格式：`{功能}Service.cs`
4. 实现 IDisposable（如需要）

```csharp
namespace ComputerCompanion.Services;

public class NewService : IDisposable
{
    // 服务实现
}
```

### 4.3 添加新视图

1. 在 `Views/` 目录创建 XAML 文件
2. 创建对应的代码后端文件
3. 确保视图名称与视图模型对应

```
Views/NewWindow.axaml      # XAML 定义
Views/NewWindow.axaml.cs   # 代码后端
```

### 4.4 添加新视图模型

1. 在 `ViewModels/` 目录创建文件
2. 继承 `ObservableObject`
3. 使用 `[ObservableProperty]` 特性

```csharp
namespace ComputerCompanion.ViewModels;

public partial class NewWindowViewModel : ObservableObject
{
    [ObservableProperty]
    private string _title = "新窗口";
}
```

## 5. 文件移动规范

### 5.1 移动前检查

| 检查项 | 说明 |
|--------|------|
| 引用关系 | 检查所有对该文件的引用 |
| 命名空间 | 更新为新目录对应的命名空间 |
| using 语句 | 更新其他文件的引用 |

### 5.2 移动步骤

1. **备份**：创建备份或提交当前更改
2. **移动文件**：使用 IDE 重构功能
3. **更新引用**：自动更新所有引用
4. **验证构建**：确保项目正常编译
5. **更新文档**：同步更新本文档

### 5.3 移动后验证

```bash
dotnet build -c Debug
```

确保无编译错误。

## 6. 文件删除规范

### 6.1 删除前检查

| 检查项 | 说明 |
|--------|------|
| 引用关系 | 确保无其他文件引用 |
| 功能完整性 | 删除不会破坏现有功能 |
| 版本控制 | 考虑是否需要保留历史 |

### 6.2 删除流程

1. **搜索引用**：确认无使用
2. **备份代码**：如有价值，保留注释或备份
3. **执行删除**：移除文件
4. **验证构建**：确保项目正常编译
5. **检查日志**：确认删除内容

## 7. 代码组织原则

### 7.1 单一职责原则

每个类应只有一个修改原因：

```
✅ 好的设计：
- HardwareMonitorService.cs：仅负责硬件监控
- SettingsService.cs：仅负责设置管理

❌ 不好的设计：
- HardwareAndSettingsService.cs：同时负责多个不相关功能
```

### 7.2 依赖方向

```
Views → ViewModels → Services → Models

不应有反向依赖！
```

### 7.3 命名空间对应

| 文件位置 | 命名空间 |
|----------|----------|
| `Models/` | `ComputerCompanion.Models` |
| `Services/` | `ComputerCompanion.Services` |
| `ViewModels/` | `ComputerCompanion.ViewModels` |
| `Views/` | `ComputerCompanion.Views` |

## 9. 常见问题处理

### 9.1 循环依赖

**问题**：Service A 依赖 Service B，Service B 又依赖 Service A

**解决**：
- 提取公共接口到新文件
- 使用依赖注入解耦
- 考虑合并相关服务

### 9.2 跨层引用

**问题**：View 直接引用 Model

**解决**：
- 通过 ViewModel 中转
- 保持 MVVM 架构层次

### 9.3 大类文件

**问题**：单个文件超过 500 行

**解决**：
- 按功能拆分为多个类
- 使用 partial class（视情况）
- 提取独立功能到新类
