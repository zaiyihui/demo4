## 2. Git 工作流程

### 2.1 分支管理策略

#### 2.1.1 主要分支

```
main (或 master)
├── develop          # 开发分支
├── release/x.x.x    # 发布分支
└── feature/*        # 功能分支
├── hotfix/*         # 热修复分支
└── bugfix/*         # 缺陷修复分支
```

#### 2.1.2 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|---------|---------|------|
| 功能分支 | `feature/功能描述-编号` | `feature/overlay-optimization-25` |
| 热修复分支 | `hotfix/问题描述-编号` | `hotfix/memory-leak-fix-30` |
| 缺陷修复分支 | `bugfix/问题描述-编号` | `bugfix/crash-on-start-31` |
| 发布分支 | `release/版本号` | `release/1.0.0` |

**命名规则**:
- 使用英文描述，使用连字符 `-` 分隔
- 避免使用中文字符
- 保持简洁，控制在 50 字符以内
- 包含相关的问题编号（如有）

### 2.2 提交信息规范

#### 2.2.1 提交信息格式

```
<类型>(<范围>): <简短描述>

[可选的正文]

[可选的脚注]
```

#### 2.2.2 提交类型

| 类型 | 描述 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(overlay): 添加FPS显示功能` |
| `fix` | 缺陷修复 | `fix(memory): 修复内存泄漏问题` |
| `docs` | 文档更新 | `docs(readme): 更新项目说明` |
| `style` | 代码格式调整（不影响功能） | `style(format): 格式化代码` |
| `refactor` | 代码重构 | `refactor(settings): 重构设置模块` |
| `perf` | 性能优化 | `perf(timer): 优化刷新频率` |
| `test` | 测试相关 | `test(viewmodel): 添加ViewModel测试` |
| `chore` | 构建/工具相关 | `chore(git): 更新.gitignore` |
| `ci` | CI/CD 配置 | `ci(pipeline): 配置自动化测试` |
| `revert` | 回滚提交 | `revert: 回滚之前的提交` |

#### 2.2.3 提交信息示例

**简单提交**:
```
feat(overlay): 添加悬浮窗位置预设功能
```

**详细提交**:
```
fix(ipc): 修复IPC消息处理线程安全问题

- 使用Volatile.Read确保线程安全
- 添加异常处理避免崩溃
- 添加详细的错误日志

关闭问题 #28
```

**中文提交示例**:
```
feat(settings): 添加颜色预设管理功能

- 实现5套专业级颜色预设方案
- 支持用户自定义预设保存
- 添加预设导入/导出功能

相关需求: #25
```

### 2.3 代码审查流程

#### 2.3.1 Pull Request 创建规范

1. **标题格式**: `[类型] 简短描述 (#issue)`
   - 示例: `[feat] 添加悬浮窗透明度和位置控制 (#25)`

2. **PR 描述模板**:
   ```markdown
   ## 概述
   [简要描述本次更改的内容和目的]

   ## 更改类型
   - [ ] 新功能 (feat)
   - [ ] 缺陷修复 (fix)
   - [ ] 文档更新 (docs)
   - [ ] 代码重构 (refactor)
   - [ ] 性能优化 (perf)
   - [ ] 测试相关 (test)
   - [ ] 其他 (chore)

   ## 更改详情
   [详细列出所有更改]

   ## 测试验证
   - [ ] 本地测试通过
   - [ ] 新增单元测试
   - [ ] 集成测试通过
   - [ ] 手动验证完成

   ## 截图/截图（如有 UI 更改）
   [添加相关截图]

   ## 相关问题
   关闭 #issue_number
   ```

#### 2.3.2 代码审查检查清单

**功能性检查**:
- [ ] 代码实现了预期的功能
- [ ] 没有引入新的 bug
- [ ] 边界条件处理正确
- [ ] 错误处理完善

**代码质量检查**:
- [ ] 代码符合项目的编码规范
- [ ] 没有重复代码
- [ ] 变量/函数命名有意义
- [ ] 注释清晰、必要

**性能检查**:
- [ ] 没有明显的性能问题
- [ ] 资源使用合理
- [ ] 没有内存泄漏

**安全性检查**:
- [ ] 没有硬编码的敏感信息
- [ ] 输入验证完善
- [ ] 权限控制正确

## 3. 文件管理规范

### 3.1 必须版本控制的文件

| 类别 | 文件类型 | 说明 |
|------|----------|------|
| 源代码 | `*.cs`, `*.axaml` | C# 源代码和 Avalonia UI 定义 |
| 项目配置 | `*.csproj`, `*.sln` | 项目和解决方案文件 |
| 测试代码 | `*Tests.cs` | 单元测试和集成测试 |
| 文档 | `*.md`, `*.json` | 项目文档和配置文件 |
| 资源 | `*.ico`, `*.png`, `*.jpg` | 图标和图片资源 |

### 3.2 禁止版本控制的文件

| 类别 | 文件类型/目录 | 原因 |
|------|---------------|------|
| 构建产物 | `bin/`, `obj/`, `Build/`, `Debug/`, `Release/` | 编译生成的文件 |
| IDE 配置 | `.vs/`, `.idea/`, `.vscode/` | IDE 特定的配置 |
| 用户文件 | `*.user`, `*.suo`, `*.rsuser` | 用户特定的设置 |
| 日志文件 | `*.log`, `logs/`, `Logs/` | 运行日志 |
| 缓存 | `*.cache`, `Cache/` | 各种缓存文件 |
| 系统文件 | `Thumbs.db`, `.DS_Store` | 操作系统生成的文件 |
| 依赖包 | `packages/` | NuGet 包（应使用 package restore） |
| 本地配置 | `appsettings.*.json` (本地) | 包含敏感信息的配置 |

### 3.3 大文件管理

- **限制**: 单个文件大小不应超过 10MB
- **大型二进制文件**: 使用 Git LFS 管理
- **第三方库**: 通过 NuGet 包管理器管理，不要直接提交

## 4. 冲突解决规范

### 4.1 预防措施

1. **频繁同步**: 每天开始工作时 `git pull origin develop`
2. **小步提交**: 频繁提交，避免大量更改
3. **代码审查**: 通过 PR 进行代码审查，及早发现问题
4. **通信**: 与团队成员保持沟通，避免同时修改同一文件

### 4.2 冲突解决步骤

#### 4.2.1 准备阶段

```bash
# 1. 确保工作区干净
git status

# 2. 如果有未提交的更改，先暂存
git stash

# 3. 拉取最新代码
git fetch origin
git pull origin develop

# 4. 恢复暂存的更改
git stash pop
```

#### 4.2.2 解决冲突

```bash
# 1. 查看冲突文件
git status

# 2. 编辑冲突文件，解决冲突标记
# <<<<<<< HEAD
# 你的代码
# =======
# 他们的代码
# >>>>>>> branch-name

# 3. 标记已解决
git add <resolved-file>

# 4. 提交（不要使用 -m 参数，让 Git 打开编辑器）
git commit
```

#### 4.2.3 解决原则

- **保持意图**: 理解每段代码的意图，选择正确的解决方案
- **沟通**: 如有疑问，与代码作者沟通
- **测试**: 解决后运行测试，确保功能正常
- **提交**: 单独提交冲突解决，不要混合其他更改

## 5. 版本标签管理

### 5.1 标签命名规范

```
v<主版本>.<次版本>.<修订版本>[-<预发布标签>]
```

**示例**:
- `v1.0.0` - 正式发布版本
- `v1.0.0-alpha` - 预发布版本
- `v1.0.0-beta.1` - 测试版本
- `v1.1.0-rc.1` - 候选发布版本

### 5.2 标签创建流程

```bash
# 创建标签
git tag -a v1.0.0 -m "正式发布版本 1.0.0"

# 推送标签到远程
git push origin v1.0.0

# 或推送所有标签
git push origin --tags
```

## 6. 日常开发流程

### 6.1 功能开发流程

```
1. 从 develop 创建功能分支
   git checkout -b feature/your-feature develop

2. 开发功能，编写测试
   git add .
   git commit -m "feat(scope): 描述"

3. 同步 develop 分支
   git fetch origin
   git rebase origin/develop

4. 推送分支，创建 PR
   git push -u origin feature/your-feature

5. 代码审查，修复问题

6. 合并到 develop
   (通过 GitHub/GitLab PR 界面合并)
```

### 6.2 缺陷修复流程

```
1. 从 develop 创建修复分支
   git checkout -b bugfix/issue-description develop

2. 修复缺陷，添加测试
   git add .
   git commit -m "fix(scope): 描述"

3. 创建 PR，审查后合并
```

### 6.3 热修复流程

```
1. 从 main 创建热修复分支
   git checkout -b hotfix/issue-description main

2. 修复并测试
   git add .
   git commit -m "fix: 紧急修复"

3. 同时合并到 main 和 develop
   git checkout main
   git merge --no-ff hotfix/issue-description
   git tag -a v1.x.x

   git checkout develop
   git merge --no-ff hotfix/issue-description
```

## 7. 最佳实践

### 7.1 提交最佳实践

1. **原子提交**: 每个提交只包含一个逻辑更改
2. **有意义的消息**: 提交信息应清晰描述更改内容
3. **频繁提交**: 尽早提交，避免大量更改带来的冲突
4. **测试先行**: TDD 流程，先写测试再实现功能
5. **审查代码**: 提交前自己先审查一遍代码

### 7.2 分支管理最佳实践

1. **保护主分支**: main/develop 分支不能直接推送
2. **功能分支短命**: 功能分支开发完成后尽快合并
3. **定期清理**: 删除已合并的分支
4. **保持同步**: 经常从主分支拉取更新

### 7.3 解决冲突最佳实践

1. **理解代码**: 仔细阅读冲突代码，理解上下文
2. **保留测试**: 确保冲突解决后测试仍然通过
3. **分步解决**: 一次只解决一个文件的冲突
4. **沟通协作**: 与相关人员沟通，避免遗漏

## 8. 常见问题与解决方案

### 8.1 提交了敏感信息怎么办？

```bash
# 从历史记录中移除
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch <file>' \
  --prune-empty --tag-name-filter cat -- --all

# 或使用 BFG Repo-Cleaner
java -jar bfg.jar --delete-files <file>
```

### 8.2 不小心提交了大文件怎么办？

```bash
# 查找大文件
git rev-list --objects --all | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | sort -k3 -n -r | head -20

# 移除大文件
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch <file>' \
  --prune-empty --tag-name-filter cat -- --all
```

### 8.3 如何撤销最后一个提交？

```bash
# 保留更改
git reset --soft HEAD~1

# 不保留更改
git reset --hard HEAD~1
```
